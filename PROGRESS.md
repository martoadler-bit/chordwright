# Chordscape — Progress

A pocket "chord ideas machine" synth for iPhone, inspired by Telepathic Instruments' Orchid (chord-generative workflow, and — as of the v2 pass below — its actual manual/workflow) — own visual identity (cream/pastel palette, flower-petal icon), not a skin clone.

**Renamed from "Bloom" (2026-07-10):** before publishing, checked whether "Bloom" would cause App Store/trademark trouble — it collides directly with Brian Eno & Peter Chilvers' long-running "Bloom" generative music app (Opal Limited, since 2008, same category, still actively sold with a 10th-anniversary edition), a real risk both for App Review's confusingly-similar-name check and for a later trademark complaint. Renamed the whole project (folders, Xcode targets, bundle id `com.dlrk.bloom`→`com.dlrk.chordscape`, every `Bloom*`/`bloom*` Swift identifier) to **Chordscape** via the same `sed`-based rename pattern used to spin up every app in this portfolio. Checked "Chordscape" (and an earlier candidate, "Harmonaut" — rejected, collides with an existing ToneBytes synth VST plugin and another electronic-instrument project of the same name) against web search first; no existing app, plugin, or trademark found in music/synth software. The app icon (flower-petal design) wasn't changed — it still reads fine as an abstract mark even though the name is no longer literally floral; worth a fresh design pass later if it ever feels like a mismatch.

## What it does

- Tap one of 7 pads → generates a full chord rooted on that pad, probabilistically extended to 7th/9th/11th/13th based on the **Complexity** knob. Same pad can return a slightly different voicing each time — the "ideas machine" surprise.
- **Key Mode** (toggle, on by default): on, each pad's chord *quality* is derived automatically from the current scale, like a real diatonic progression. Off, quality instead comes from an explicit **Dim/Min/Maj/Sus** picker — the pad still just picks a root (see Known simplifications), but the quality is now yours to choose regardless of what the scale would diatonically put there, mirroring Orchid's chord-type buttons.
- Extensions are quality-aware, not just diatonic: chords with a major 3rd (major/dominant) get the 11th raised to a **♯11** (a diatonic natural 11 sits a half-step above a major 3rd — the classic "avoid note" clash); chords with a minor 3rd keep a natural 11 (no clash there). Shown on-screen as e.g. "C(♯11)" vs "Dm11".
- **Voicing knob**: cycles the chord through inversions in either direction (lowest note up an octave per step, or highest note down) — Orchid's "invert chords... to help it better work with a particular sound." When the bass note isn't the root anymore, the on-screen label grows a slash, e.g. "C/E".
- **Performance mode** picker: **Sustain** (all notes together, as before), **Arp** (cycles one chord tone at a time, eighth notes at the current tempo, for as long as the pad is held), **Strum** (notes fire staggered low-to-high over ~20ms, then ring together, like a pick across strings), **Slop** (notes fire with small random timing jitter instead of a fixed stagger — a "human hands" feel). Orchid ships 5 modes (also Pattern and Harp); these 3 are the ones that don't need a preset-pattern library or a dedicated glissando voice-stealing scheme to be worth having.
- Root note + scale (Major, Minor, Dorian, Mixolydian, Harmonic Minor, Whole Tone) pickers.
- 6 synth engines: **Subtractive** (2 detuned saws → state-variable lowpass), **FM** (2-operator, fixed 2:1 ratio, bright/bell-ish), **Reed** (electric-piano-style FM bark — modulator has its own fast, independent decay envelope separate from the main amp envelope), **Square** (single PWM pulse wave, duty cycle sweeping slowly on its own, through the same filter as Subtractive — hollower/reedier), **Bell** (2-op FM at a fixed *inharmonic* ratio — metallic/glass partials instead of FM's musical ones), **Pluck** (Karplus-Strong plucked string — a noise burst re-circulated through a short damped delay line, the only non-oscillator engine, decays naturally like a real string).
- **Patches**: save/load full instrument snapshots (root, scale, complexity, voicing, Key Mode + quality, performance mode, engine, tone/decay/ambience/bass/glide/tempo — not the loop's recorded content, that's session data not instrument sound). 13 factory patches ship built in (several genuinely different patches per engine, not a mechanical one-each); save your own from the current knob/picker state under any name. Reachable by tapping the current patch's name under the app title.
- **Filter mod**: drag straight up from a held pad to brighten the sound — a global filter-cutoff sweep (all voices share one cutoff, so this isn't per-note expression) that snaps back to the Tone knob's own value the instant you lift. Works on the same touch as glissando, so a diagonal drag glides across pads and brightens at once.
- **Finger glow**: a soft radial glow in the pad's own color follows your finger while it's down, growing bigger/brighter as the filter-mod gesture engages — visual feedback for the otherwise-invisible cutoff sweep, and just decoration the rest of the time.
- **Dark mode**: a sun/moon toggle in the header switches the whole cream/pastel palette to a dark variant of itself (same hues, just deeper background/panel and brighter pad colors for contrast) — persists across launches, deliberately separate from Patches (it's a display preference, not part of "how the instrument sounds").
- **Dedicated bass voice**: a separate always-mono sub-bass note (fixed round subtractive tone, own envelope) follows whichever pad is currently held, roughly 2 octaves below the chord — independent of the 3-engine picker, own **Bass** level knob, mirrors Orchid's separate bass synth engine.
- **Glissando**: press a pad, then drag your finger into a neighboring pad without lifting — the bass **always** slides smoothly toward the new root (own **Glide** knob, 0–0.5s), independent of performance mode. What happens to the chord voices depends on the mode: **Sustain** slides their pitch too (a true glide); **Arp** doesn't slide — instead it keeps ticking at its own steady rhythm and just swaps which chord it's drawing notes from, so crossing a pad boundary doesn't retrigger anything, the very next arp tick simply plays from whatever chord is currently under the finger; **Strum/Slop** do a clean release-then-fresh-trigger on each crossing (neither has a coherent "glide" reading). A plain tap that never leaves its pad behaves exactly as before in all modes — this only kicks in once a drag actually crosses into another pad.
- 6 knobs in a 3-column grid (2 rows): Complexity, Voicing, Tone, Decay, Ambience, Glide; 2 more (Bass, Tempo) live in the transport row.
- 16-step loop (one bar of 4/4 in sixteenths): press Record, play pads, each press gets written into whichever step is currently playing. Loop always runs once Play is on; steps hold their chord "legato" until the next step (empty or not) takes over. Loop playback always uses plain sustained triggering regardless of which performance mode was live when a step was recorded — the mode is a live-play gesture, not part of the loop's stored data.
- **Panic button** (small "stop" icon in the header): immediately silences every currently-sounding voice and resets all local "what's held" bookkeeping — a recovery escape hatch if a note ever gets stuck.
- Root/Scale/Key Mode/Quality/Engine/Performance Mode live on the main screen (briefly moved into a Settings sheet, user asked for that reverted the same day — see the dated note below). Only **Patches** stays as its own menu.
- Layout order puts the pad row **last** — it's the "keyboard," everything else (pickers, knobs, transport) reads as controls sitting above the playing surface rather than interleaved with it.

## Architecture

- `Core/Music/ChordEngine.swift` — scale tables + `generate(...)` (diatonic path, quality auto-derived) and `generateChromatic(...)` (Key-Mode-off path, quality from `ChordQualityOverride`'s hand-authored interval tables). `applyVoicing(_:steps:)` cycles inversions by repeatedly moving the current lowest/highest note an octave; `slashLabel(...)` appends "/bassNote" when voicing puts a non-root note on the bottom.
- `Core/Music/PerformanceMode.swift` — the Sustain/Arp/Strum/Slop enum (UI-facing; the actual scheduling logic lives in the engine, see below).
- `Core/DSP/SynthVoice.swift` — per-voice oscillator/filter/envelope math for all 6 engines (3 new: `renderSquare` reuses the same SVF/`filterLow`/`filterBand` state as `renderSubtractive`, safe since only one engine renders pool-wide at a time; `renderBell` is `renderFM`'s code shape with the modulator ratio changed from 2:1 to a fixed 3.01:1 inharmonic ratio; `renderPluck` needs its own per-voice state — `pluckBuffer`/`pluckIndex`/`pluckNeedsInit` — since Karplus-Strong needs an actual delay line, not just phase accumulators. `pluckNeedsInit` is a real flag set in `noteOn` rather than inferred from "does the buffer length match the current frequency", specifically so a glide's continuously-changing frequency doesn't re-excite/re-pluck the string every single sample — it only gets plucked fresh on an actual new note, same as a real string doesn't retune smoothly either). `PolySynth.swift` — 24-voice pool. `trigger(midiNotes:groupID:)` replaces whatever a group holds (used for plain Sustain and loop playback); `addNote(_:groupID:)` adds one voice to a group *without* releasing what's already there (used to build a chord up incrementally for Strum/Slop/Arp). `groupID` tags let "release whatever pad 3 is holding" work without tracking pitches by hand; voice stealing picks the quietest voice when the pool is full. Each voice also tracks a `currentPitch`/`targetPitch` pair — normally pinned equal (instant), but `glideTo(midi:groupID:glideSeconds:)` sets them apart and lets `render()` linearly interpolate between them sample-by-sample without touching phase or envelope, which is what makes a glissando note keep ringing through the slide instead of clicking/restarting. `PolySynth.glideChord(fromGroupID:toGroupID:midiNotes:glideSeconds:)` pairs the old chord's voices (sorted by current pitch) against the new chord's notes (sorted ascending) index-wise and retags them. Voice count stays locked to whatever was already sounding: a bigger new chord's extra notes are simply not played (a fresh-attack voice popping in mid-slide would read as a pop, not a glide); a smaller new chord releases the extra old voices normally (letting go doesn't have that problem). `SynthVoice.retag(to:)`/`PolySynth.retagGroup(from:to:)` change a voice's groupID tag without touching pitch/phase/envelope at all — used by the Arp retarget path below, not by the pitch-glide path.
- `Core/DSP/Effects.swift` — `SimpleDelay`/`SimpleReverb` ported from Modula's `DelayNode`/`ReverbNode` recipes; `SimpleChorus` is new (LFO-modulated short delay read at 2 phase offsets for stereo width — the main thing that keeps a single mono voice from sounding thin).
- `Core/Engine/ChordscapeAudioEngine.swift` — single `AVAudioSourceNode` render callback owns the chord synth, the dedicated bass `SynthVoice`, the effects chain, the loop clock, and the performance-mode scheduler. **Threading rule, mirrors Modula's discipline:** `PolySynth`/`SynthVoice` are touched only from the audio thread. Pad taps enqueue a `SynthCommand` under a lock (`.trigger`/`.strum`/`.slop`/`.arpStart`/`.release`/`.glideRetag`/`.arpRetarget`/`.glideNote`); the render callback drains that queue each block, and everything the drained command sets up from there (`scheduledNotes` for Strum/Slop's staggered onsets, `activeArps` for the arpeggiator's stepping) is itself audio-thread-only state, never touched from the main thread. All of it — the loop clock, the strum/slop stagger, the arp step — is block-quantized (checked once per render callback against a running `audioSampleClock`, not per-sample) — jitter bounded by the audio buffer size (typically ~10ms), an accepted simplicity tradeoff, not what you'd want for a hi-hat but fine for chord-paced material. The bass's `.trigger`/`.release`/`.glideNote` commands are routed to the dedicated `bassVoice` directly (`applyBass(_:)`) rather than through the shared `synth` pool — **this fixes a real bug found while wiring glide**: the bass had originally been going through `synth.trigger(...)` like a chord voice, which meant it played with the main chord's tone/cutoff/decay instead of `bassParams`' fixed round tone, and the actual dedicated `bassVoice` instance sat silently unused the whole time (its `noteOn` was never called). Confirmed fixed by re-reading the command-routing path line by line, not by ear (no Simulator audio here). Bass has its own top-level API (`triggerBass`/`releaseBass`/`glideBass`) fully decoupled from `padDown`/`padUp`/`glideChordVoices`/`retargetArp` — it never took a `bassMIDI`/`releaseBass` parameter through those anymore, so nothing about the chord voices' performance mode can leak into (or gate) what the bass does. `.arpRetarget` (in `apply(_:)`) moves an in-flight `ArpState` from the old pad's dictionary key to the new pad's, preserving `index`/`nextFireSample` untouched (so the arp's own tick rate is never disturbed by a crossing) and retagging the currently-sounding voice via `PolySynth.retagGroup` so it stays findable by a later release.
- `ViewModel/ChordscapeViewModel.swift` — owns chord generation + voicing math (pure, no audio-thread concerns); the engine only ever receives final notes. Tracks which pads are physically held (`heldPadRoots`) so the monophonic bass note only lets go once *every* pad is up, independent of the polyphonic chord voices. `glissandoChanged(originPad:translationX:translationY:padWidth:)`/`glissandoEnded(originPad:)` track one active pad per drag-origin (keyed by `originPad`, not a single shared variable) so two simultaneous drags from two different fingers don't clobber each other; the first `onChanged` call of a drag (translation ≈ 0) behaves exactly like a plain tap-down, so simple taps are unaffected. `performGlide(from:to:)` branches on `performanceMode` to decide what a pad-crossing does to the chord voices (Sustain glides, Arp retargets, Strum/Slop release-and-retrigger) completely independently of the bass, which always glides via `bassOwnerPad` tracking regardless of which of those three branches ran. `padModulationChanged(translationY:)`/`padModulationEnded()` (called from the same `glissandoChanged`/`glissandoEnded`, since it's the same touch) read the drag's vertical component for the filter-mod gesture — `engine.setCutoff(...)` is just called directly with a value swept between the Tone knob's own value and the max, since `VoiceParams.cutoff` is already a plain cross-thread setter with no per-note concept; no new engine API was needed.
- `Views/` — SwiftUI, cream/pastel theme, now with a toggleable dark variant (was fixed-light-only for most of this app's life; see the dated note below). `SegmentedIconPicker<T: PickerOption>` is one generic picker component reused for the engine, quality, and performance-mode rows (was 3 near-identical views, generalized once the pattern repeated a third time). `ContentView` observes both `ChordscapeViewModel` (`@StateObject`) and `ChordscapeAudioEngine` (`@ObservedObject`, pulled out of the view model in `init()`) since nested `ObservableObject`s don't auto-propagate `@Published` changes through SwiftUI. Now wrapped in a `ScrollView` — the control count grew enough (2 more picker rows, more knobs) that a fixed-height iPhone SE layout could no longer be assumed to fit. `padRow` wraps the 7 pads in a `GeometryReader` to compute a shared `padWidth`, but gives each pad its **own** `DragGesture` (not one gesture over the whole row) — SwiftUI keeps delivering a touch's updates to whichever pad captured it even once the finger has visually moved past that pad's bounds, so per-pad gestures are enough to detect a glissando while still letting two different fingers hold two different pads independently. `PadView` itself went from owning its own press gesture + `@State` to being purely presentational (`isActive: Bool` passed in), since gesture handling now has to live at the row level.
- `Views/Components/GlowView.swift` — the finger-following glow. **First version didn't render at all** (user report: "no anda el brillito") — it used `.coordinateSpace(name: "padRow")` on the `GeometryReader` plus `drag.location` in that named space, with the glow positioned via `.position(point)` from a `ZStack` *sibling* of the `GeometryReader`. That's a fragile SwiftUI pattern (a named-coordinate-space handshake across a `GeometryReader`/sibling-`ZStack` boundary) and apparently didn't actually work. Fixed by restructuring so the `ZStack` is *inside* the `GeometryReader` (not the reverse) and computing the glow's position from the exact same `drag.translation` already used for glissando/filter-mod, added to pad `i`'s own known center (`CGFloat(i) * (padWidth + spacing) + padWidth / 2`, `geo.size.height / 2`) — both the gesture and the glow now read from the same `geo`, so there's no cross-view coordinate-space matching to get wrong. Trade-off: this assumes the touch started near the pad's center rather than tracking the exact touch-down point, a fine approximation for decoration. `ContentView` tracks one `CGPoint` per active touch (`glowPoints: [Int: CGPoint]`, keyed by drag-origin pad, same per-touch pattern the view model already uses). `GlowView.intensity` reads `ChordscapeViewModel.modulationAmount` (`@Published Float`, updated alongside the actual cutoff sweep in `padModulationChanged`/`padModulationEnded`) so the glow grows/brightens in sync with the filter-mod gesture.
- **Layout regression fixed (2026-07-11):** user reported everything looked scaled up and nearly touching the screen edges, less clean than the original 4-knob layout. Root cause, found by re-reading the code (no Simulator here to see it directly): `ChordscapeKnob`'s label `Text` had no width constraint or `lineLimit`, so labels like "Complexity"/"Ambience" rendered wider than the 54pt knob circle itself; a straight `HStack` of 5 (then 6, once Glide was added) of these plus a 3-knob transport row pushed total row width past comfortable screen bounds, collapsing the `Spacer()`s toward zero and crowding everything against the padding. Fixed two ways: (1) `ChordscapeKnob`'s label now has `.lineLimit(1)`, `.minimumScaleFactor(0.7)`, and `.frame(maxWidth: diameter + 16)` so a long label shrinks instead of ballooning the cell width; (2) `knobRow` became a `LazyVGrid` (3 columns × 2 rows: Complexity/Voicing/Tone/Decay/Ambience/Glide) instead of one long `HStack`, giving each knob a fixed, predictable share of the width regardless of screen size; Glide moved out of the transport row (back to just Bass + Tempo there, matching the original pre-v2 proportions).
- **Vertical compression pass (2026-07-11), then reverted the same day:** user reported the whole layout didn't fit on one screen without scrolling, especially with Key Mode off. Tightened spacing/padding/knob size throughout and combined Engine+Performance into one row. User then said the *previous* look (the 3×2 knob grid at full size, separate picker rows, 110pt pad row — i.e. the state right after the "Layout regression fixed" pass above) was better and asked to go back. Reverted every value from the compression pass (`VStack` spacing 10→16, outer padding 14→20, chord label font 28→34 + padding restored, knob diameter 48→54, knob grid spacing 12→22, pad row height 88→110, Engine/Performance pickers un-combined back into two separate rows). Net effect: the 3×2 grid fix stays (that one was wanted), but nothing was tightened further — so the not-fitting-without-scroll complaint is back too, traded off deliberately in favor of the roomier look. `ScrollView` remains the fallback for whatever doesn't fit.
- `Core/Patch/Patch.swift` — `Codable` snapshot struct of every tone/generative `@Published` property (not the loop) + `FactoryPatches.all` (7 hand-written presets, one per engine). All the enums it references (`MusicalScale`, `ChordQualityOverride`, `PerformanceMode`, `SynthEngineType`) picked up `Codable` conformance for this — free via Swift's automatic synthesis for simple raw-value enums, no custom `Codable` code needed anywhere. `Core/Patch/PatchStore.swift` — one JSON file per user patch in `Documents/Patches/<uuid>.json`, same recipe as Modula's `PatchStore`. `ChordscapeViewModel.loadPatch(_:)` just assigns every `@Published` property from the patch (the ones with `didSet` push to the engine exactly like a knob drag would); `saveCurrentPatch(as:)` builds a `Patch` from current state and persists+appends it; `userPatches` loads from `PatchStore` once in `start()`.
- `Views/PatchBrowserView.swift` — factory + user patch list, save-as text field, tap a row to load+dismiss, trash icon to delete user patches only (factory patches aren't deletable). Plain `ScrollView`+`VStack` rather than SwiftUI's `Form`/`List` — those bring their own default (gray/white) list styling that would clash with the app's custom cream/pastel palette. (A `SettingsSheetView` briefly existed holding Root/Scale/Key Mode/Quality/Engine/Performance — deleted the same day once the user asked for those back on the main screen; see below.)
- `Core/Engine/MIDIManager.swift` — ported straight from Modula's `MIDIManager` (same Core MIDI client/port setup, same packet-list-pointer-not-a-local-copy parsing gotcha, no explicit `CoreMIDI.framework` linking needed in the pbxproj — same as Modula, confirmed by a clean `xcodebuild`). Auto-connects to every available source (USB/Bluetooth/virtual) and forwards Note On/Off to `ChordscapeViewModel.externalNoteOn(_:)`/`externalNoteOff(_:)`. Since a MIDI controller has 12 chromatic keys but Chordscape only has 7 pads, each incoming note snaps to whichever pad's root sits closest in pitch class (circular distance, octave ignored — pads already always render at a fixed octave regardless of where you tap, so this matches existing behavior). Ref-counted per pad (`midiPadHoldCounts`) so two colliding notes snapped to the same pad don't let releasing one cut a chord the other is still holding. Once snapped, a MIDI note plays through the exact same `padDown`/`padUp` path as an on-screen tap — same chord generation, voicing, performance mode, everything (MIDI input does not currently trigger a glissando — see Known simplifications). `ContentView`'s header shows a small pill with the connected source's name when one is active.

- **Dark mode added (2026-07-11):** user asked, as an open question alongside the finger-glow idea, whether to add a dark mode toggle — flagged it as more work than the glow (needs a genuinely designed second palette, not a color inversion) and suggested doing it as a separate pass; user agreed, then asked for it once the glow was working. `Colors.swift` was rewritten so every `Color.chordscapeXxx` is built via a new `Color.dynamic(light:dark:)` helper — bridges through `UIColor { traits in traits.userInterfaceStyle == .dark ? ... : ... }`, which SwiftUI resolves automatically against whatever `.preferredColorScheme` is active. This meant **zero changes at any of the ~10 view files** that already reference these colors — only `Colors.swift` itself changed. `ContentView` gained `@AppStorage("chordscape.isDarkMode")` (persists across launches, since it's a display preference, not instrument state — deliberately not part of `Patch`) and a sun/moon toggle button in the header driving `.preferredColorScheme(isDarkMode ? .dark : .light)`. Dark hues keep the same 7-pad-color family, just brightened slightly for contrast against a dark background rather than simply inverted.

## Known simplifications (deliberately cut)

- **Key Mode off is not full chromatic freedom**: the pad row still only offers the current scale's 7 roots — Key Mode off changes *quality* (Dim/Min/Maj/Sus instead of diatonic auto-quality), not *which roots* are available. Real 12-tone chromatic root freedom (a la Orchid's actual keyboard) would need a different pad layout and was deferred.
- No velocity/pressure sensitivity — every pad triggers at fixed volume.
- Loop is fixed at 16 steps / 1 bar, no bar-length or step-resolution options.
- Reverb is mono-in→mono-processed then split to both channels (chorus after it is what actually gives stereo width); not a true stereo reverb.
- Patches capture tone/generative settings only — loop content isn't part of a patch and still resets on relaunch (deliberate: a patch is "how the instrument sounds," not "what you played").
- MIDI **input** only (auto-connects to any source, snapped to the nearest pad) — no MIDI **output** (Orchid can drive external synths over DIN MIDI; Chordscape can't send anything out), no MIDI Learn, no CC/pitch bend, no velocity sensitivity (a Note On's velocity byte is read but ignored, consistent with pads having no velocity either).
- No Pattern or Harp performance modes (Orchid's other 2) — Pattern needs a preset-rhythm library, Harp is closer to a continuous rolling glissando; both deferred as separate, larger pieces of work.
- Whole Tone scale only has 6 degrees but the pad row is fixed at 7 — the 7th pad repeats the same pitch class as the 1st (no octave-up correction). Harmless quirk, not worth special-casing for one scale.
- Complexity's extension rolls happen fresh on every pad press (no per-pad "memory" of what it last played), so building a stable progression across a loop relies on the Complexity knob being low/predictable rather than the app remembering anything.
- Bass note follows "whichever pad was pressed most recently, of whatever's still held" rather than true last-note-priority re-triggering if you release one held pad while another is still down — a deliberate simplification to avoid extra held-note-ordering bookkeeping.
- **Real stuck-note bug fixed (2026-07-11):** user reported notes sometimes never stopping when changing options while playing. Found one concrete, reproducible cause by code trace: `performGlide` used to send a bass-retarget command on *every* glissando step unconditionally, regardless of whether the gliding pad was the one actually driving the bass. If pad A was held independently (e.g. a second finger) while a glissando dragged across pad B→C, the bass would get silently hijacked onto C's pitch, and releasing C wouldn't free it (`heldPadRoots` still had A in it, so `releaseBass` never fired) — an audible stuck bass note. Fixed by tracking `bassOwnerPad` explicitly and only retargeting/releasing bass through whichever pad currently owns it. This is a real, traced-and-fixed bug, but it's also very plausibly not the *only* possible stuck-note path — a global **panic button** (see above) was added as a safety net regardless, since interactive testing isn't possible in this environment and other causes (e.g. some SwiftUI gesture-cancellation edge case) can't be fully ruled out by code review alone.
- Glissando is a single-touch-drag-only gesture — MIDI input doesn't trigger it (a MIDI note-to-note transition always uses a fresh attack). If two separate glissando drags (two fingers) converge on the same target pad, releasing one of them will cut that pad's sound even though the other finger is still logically "holding" it — accepted rather than adding cross-drag ref-counting on top of the existing per-drag tracking.
- Glissando's Glide knob only sets the rate for the bass and Sustain-mode pitch-slides — Arp's retarget and Strum/Slop's release-and-retrigger don't have a "slide duration" to speak of, so the knob doesn't affect how a crossing feels in those two modes.
- **Filter mod is global, not per-note**: `VoiceParams.cutoff` is one value shared by every currently-sounding voice, so dragging up brightens *everything* ringing, not just the notes under that finger — real per-note expression (MPE-style) would need cutoff to move into `SynthVoice` itself, a bigger change deferred for now. With two fingers both modulating on different pads, whichever `onChanged` fires last wins, and either lifting resets the shared cutoff back to the Tone knob's value even if the other finger is still dragging — accepted since the feature is clearly aimed at the single-finger case.
- Filter mod has no visual feedback (the Tone knob doesn't show the live modulated value moving) — audio-only for now.
- Glissando voice count is locked to whatever was already sounding when the drag started: sliding into a chord with *more* notes than that just drops the extras (you hear the lowest N of the target chord, N = however many voices were already ringing), rather than adding them mid-slide. A deliberate tradeoff — dropping notes is less jarring than a fresh-attack pop cutting across an otherwise-smooth glide. (Applies to Sustain's pitch-slide only — Arp's retarget always uses the full target chord since it's not pairing/gliding individual voices.)
- **Glissando doesn't pitch-glide the chord voices in Arp/Strum/Slop** — only Sustain gets a true slide, and the bass always glides regardless of mode (see below). Strum/Slop do a clean release-then-fresh-trigger on each crossing (neither a strum's one-shot staggered onset nor a re-triggering arp has a coherent "portamento" reading).
- **Real stuck-note bug #2, fixed, then refined into a feature per user feedback (2026-07-11):** first fix made Arp/Strum/Slop all do release-then-fresh-trigger on a crossing (see the git-history-equivalent note in memory) — this stopped the leak (an arp left running forever in the background after gliding away, `.glideRetag` never having cleared `activeArps`) but also meant Arp's own tick rate reset on every pad crossing, and the bass (routed through the same "just re-trigger" path for non-Sustain modes) stopped gliding too. User asked for both: bass should always glide regardless of mode, and Arp specifically should keep ticking at its own steady rate rather than restarting on every crossing, playing whichever chord is currently under the finger at each tick. Implemented via `.arpRetarget` (moves the `ArpState` to the new pad's dictionary key, `index`/`nextFireSample` completely untouched, retags the currently-sounding voice via `SynthVoice.retag`/`PolySynth.retagGroup` so a later release still finds it) and by fully decoupling the bass's engine API (`triggerBass`/`releaseBass`/`glideBass`) from `padDown`/`padUp`/`glideChordVoices`/`retargetArp` so no chord-mode branching can gate what the bass does.
